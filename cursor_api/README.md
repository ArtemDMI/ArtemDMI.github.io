# file_agent — перевод больших текстовых файлов

CLI-пайплайн для перевода больших текстов через параллельные локальные запуски Cursor Agent модели **composer-2.5**.

Все команды ниже выполняются **из каталога `cursor_api`**.

**ВАЖНО: всегда передавайте в CLI полный абсолютный путь к файлу (`C:\...`) и не используйте относительные пути.**
Это убирает двусмысленность при резолвинге пути и не тратит лишнюю попытку запуска на ошибку вида `Source file not found`.

## Быстрый старт

### Перевод (`-t`)

Нормализует исходный файл, для SRT-подобных субтитров удаляет пустые строки, номера и тайминги, делит текст на части, переводит каждую часть отдельным агентом и проверяет длину перевода.
Перед стартом перевода на Windows пайплайн автоматически останавливает все уже запущенные процессы `cursor-sdk-bridge`, оставшиеся от прошлых запусков.
До запуска агентов CLI печатает примерное время ожидания по числу частей, паузам между стартами и базовому замеру одной части.

```bash
python -m file_agent run "C:\PROJECTS\ArtemDMI.github.io\sources\15\test.txt" -t "This story is about Rachel, a woman traveling across the country with Jake, a man, and Emily, a woman."
```

### Сборка после ручной правки (`-merge`)

Собирает исходный файл и части `S001`, `S002`, … обратно в один файл.

```bash
python -m file_agent run "C:\PROJECTS\ArtemDMI.github.io\sources\15\test.txt" -merge
```

## Как устроены части на диске

После `-t`:

- **Часть 1** остаётся в исходном файле (`sources/15/test.txt`).
- **Части 2, 3, …** появляются рядом с суффиксами: `testS001.txt`, `testS002.txt`, …

Пример для `sources/15/test.txt`:

| Файл | Содержимое |
|------|------------|
| `sources/15/test.txt` | первая часть |
| `sources/15/testS001.txt` | вторая часть |
| `sources/15/testS002.txt` | третья часть |

Разбиение идёт по границам предложений после нормализации (одно предложение на строку).
Если в тексте нет знаков конца предложения (`.!?…`) или нормализация схлопнула текст в несколько гигантских строк, пайплайн не считает лимит по строкам: такие файлы режутся только по бюджету около **12000** символов на часть, по возможности по пробелам.

## Промпт перевода

Основная инструкция перевода хранится в файле:

`cursor_api/file_agent/prompt.md`

Для режима `-t` после флага нужно передать короткий **английский ASCII-контекст** истории. Он подставляется перед `prompt.md` для каждого агента, чтобы части, которые переводятся отдельно, не путали имена, род, пол персонажей, отношения и точку зрения рассказчика.

Если это рассказ от лица персонажа, обязательно укажите рассказчика и его пол. Если не указано иное, пишите в контексте: `Unless stated otherwise, the story is told by Brenda, a woman.`

Формат:

```bash
python -m file_agent run "C:\PROJECTS\ArtemDMI.github.io\sources\15\test.txt" -t "This story is about Rachel, a woman traveling across the country with Jake, a man, and Emily, a woman."
```

Если SDK не поддерживает system prompt для корневого агента, команда `-t` завершится **явной ошибкой** — нужно согласовать изменение контракта; автоматической подстановки промпта в user-сообщение нет.

## Лимиты и проверки

| Параметр | Значение |
|----------|----------|
| Строк на часть (после нормализации) | до **300** |
| Символов на часть (обычный текст) | около **10000** |
| Символов на часть (если нет `.!?…`) | около **12000** |
| Попыток на часть | **3** |
| Таймаут одной попытки | **180** секунд |
| Пауза между стартами частей | **2.2** секунды |
| Модель | только **composer-2.5** |
| Очистка перед `-t` | остановка уже запущенных `cursor-sdk-bridge` на Windows |
| Базовый замер для оценки | **~156 секунд** на 1 часть около **10000** символов |

После успешного перевода каждой части CLI сравнивает длину **символов без пробелов** исходной части и перевода. Допустимое отношение: **60%–140%** (0.60–1.40).

Оценка ожидания перед стартом считается так:

`примерно = замер_одной_части + (число_частей - 1) * 2.2s`

Это оценка нормального успешного прогона без повторных попыток. Если части падают и уходят на повторы, фактическое время может быть больше вплоть до лимита `3 x 180s` на проблемную часть.

## Сводка и коды завершения

Во время `-t` в stdout печатается короткий прогресс по мере завершения частей:

```text
13 - done.
14 - fail.
```

В конце `-t` печатается краткая сводка: сколько частей всего, сколько успешно, список проблемных файлов с причиной, оценка ожидания и фактическая длительность.

- Код **0** — все части переведены и прошли проверку длины.
- Код **не 0** — была хотя бы одна ошибка.

### Частичные ошибки

Если одна или несколько частей упали (агент, таймаут, проверка длины), **остальные успешные части сохраняются на диске**. Общий процесс всё равно завершается ошибкой. Откройте проблемные файлы вручную, исправьте их, при необходимости перезапустите перевод только для нужных частей или соберите файл через `-merge`.

## Режим `-merge`

1. Находит исходный файл и подряд идущие части `S001`, `S002`, …
2. Читает их в UTF-8 в порядке номеров.
3. Склеивает через **две переводы строки** (`\n\n` между частями).
4. Перезаписывает **только** исходный файл результатом.
5. Удаляет суффиксные файлы (`testS001.txt`, …) после успешной сборки.

Если в нумерации есть разрыв (например, есть `S001` и `S003`, но нет `S002`), merge завершится ошибкой.

В конце печатается сводка: сколько файлов собрано и куда записан результат.

## Первый реальный перевод (вручную)

Автоматические тесты в репозитории **не** запускают перевод на ваших файлах в `sources/*`. Первый прогон с реальным API делаете **вы сами** на выбранном файле.

1. Выберите файл и сразу возьмите его **полный абсолютный путь**, например `C:\PROJECTS\ArtemDMI.github.io\sources\15\test.txt`.
2. Из каталога `cursor_api` запустите:
   ```bash
   python -m file_agent run "C:\PROJECTS\ArtemDMI.github.io\sources\15\test.txt" -t "This story is about Rachel, a woman traveling across the country with Jake, a man, and Emily, a woman."
   ```
3. Если CLI завершится ошибкой про **неподдерживаемый system prompt** в SDK — **остановитесь** и не продолжайте перевод. Это штатное поведение: без согласованного изменения контракта пайплайн не подставляет `prompt.md` в user-сообщение (см. раздел «Промпт перевода»).
4. При коде **0** вручную просмотрите исходный файл и все части `testS001.txt`, `testS002.txt`, … рядом с ним. При необходимости отредактируйте их в редакторе.
5. Когда части устраивают, соберите итоговый файл:
   ```bash
   python -m file_agent run "C:\PROJECTS\ArtemDMI.github.io\sources\15\test.txt" -merge
   ```

## Чего нет в пайплайне

Явно **не используются**:

- резервные копии (backup);
- fallback на другую модель, другой ключ, другой workspace или другой prompt-файл;
- скрытые режимы «на всякий случай».

Ошибка остаётся ошибкой — CLI не пытается угадать другой путь.

## API-ключ

Ключ читается из **внешнего текстового файла**; путь задан в `file_agent/keys.py`.

**`keys.py` менять не нужно**, если вы не переносите файл ключа на диске. Ключ не хранится в репозитории и не передаётся через переменные окружения CLI.
