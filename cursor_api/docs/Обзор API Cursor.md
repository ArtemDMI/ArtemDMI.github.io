## Аутентификация

Все API Cursor используют базовую аутентификацию. API облачных агентов также поддерживает Bearer-токены — выберите вариант, который удобнее для вашего HTTP-клиента.

### Базовая аутентификация

Используйте свой API‑ключ в качестве имени пользователя при базовой аутентификации (пароль оставьте пустым):

```
curl https://api.cursor.com/teams/members \
  -u ВАШ_API_КЛЮЧ:
```

Или задайте заголовок Authorization напрямую:

```
Authorization: Basic {base64_encode('YOUR_API_KEY:')}
```

### Bearer-аутентификация (API облачных агентов)

[API облачных агентов](/docs/cloud-agent/api/endpoints) также поддерживает заголовки `Authorization: Bearer <key>`. Обе схемы работают одинаково — используйте ту, которая удобнее для вашего HTTP-клиента:

```
curl https://api.cursor.com/v1/me \
  -H "Authorization: Bearer YOUR_API_KEY"
```

### Создание ключей API

Ключи API создаются в настройках команды. Создавать и управлять ключами API могут только администраторы команды.

#### Admin API & API отслеживания ИИ-кода

1. Перейдите в **cursor.com/dashboard** → вкладка **Настройки** → **Дополнительно** → **Ключи Admin API**
2. Нажмите **Создать новый API-ключ**
3. Задайте ключу понятное описательное имя (например, "Интеграция с дашбордом использования")
4. Сразу скопируйте сгенерированный ключ — больше он не будет показан

Формат ключа: `crsr_xxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxx`

Требуемая область действия: `admin:*`

#### Analytics API

Создайте API-ключ на [странице настроек команды](https://cursor.com/settings).

#### API облачных агентов

Создайте пользовательский API-ключ в [Cursor Dashboard → API-ключи](https://cursor.com/dashboard/api) или используйте [API-ключ сервисного аккаунта](/docs/account/enterprise/service-accounts) в настройках команды.

API-ключи привязаны к вашей организации и видны всем администраторам. На ключи не влияет статус аккаунта их первоначального создателя.

## Кэширование

Ряд API поддерживает HTTP‑кэширование с ETag для снижения трафика и повышения производительности.

### Поддерживаемые API

- **Analytics API**: Все конечные точки (и на уровне команды, и по пользователям) поддерживают HTTP-кэширование
- **AI Code Tracking API**: Конечные точки поддерживают HTTP-кэширование

### Как работает кэширование

1. **Первичный запрос**: Выполните запрос к любому поддерживаемому endpoint’у
2. **Ответ содержит ETag**: API возвращает заголовок `ETag` в ответе
3. **Последующие запросы**: Укажите значение `ETag` в заголовке `If-None-Match`
4. **304 Not Modified**: Если данные не изменились, вы получите ответ `304 Not Modified` без тела

### Пример

```
# Первоначальный запрос
curl -X GET "https://api.cursor.com/analytics/team/dau" \
  -H "Authorization: Bearer YOUR_API_KEY" \
  -D headers.txt

# Ответ включает: ETag: "abc123xyz"

# Последующий запрос с ETag
curl -X GET "https://api.cursor.com/analytics/team/dau" \
  -H "Authorization: Bearer YOUR_API_KEY" \
  -H "If-None-Match: \"abc123xyz\""

# Возвращает 304 Not Modified, если данные не изменились
```

### Время кеширования

- Время кеша: 15 минут (`Cache-Control: public, max-age=900`)
- Ответы содержат заголовок `ETag`
- Добавляйте заголовок `If-None-Match` в последующие запросы, чтобы получать `304 Not Modified`, если данные не изменились

### Преимущества

- **Меньше трафика**: ответы 304 не содержат тела
- **Быстрее ответы**: не требуется обрабатывать неизменённые данные
- **Не влияет на лимиты**: ответы 304 не расходуют квоту
- **Лучшая производительность**: особенно полезно для часто опрашиваемых конечных точек

## Рекомендации по лучшим практикам

### 1. Реализуйте экспоненциальный бэкофф

Когда вы получаете ответ 429, подождите перед повторной попыткой, увеличивая задержку:

```
import time
import requests

def make_request_with_backoff(url, headers, max_retries=5):
    for attempt in range(max_retries):
        response = requests.get(url, headers=headers)
        
        if response.status_code == 429:
            # Exponential backoff: 1s, 2s, 4s, 8s, 16s
            wait_time = 2 ** attempt
            print(f"Rate limited. Waiting {wait_time}s before retry...")
            time.sleep(wait_time)
            continue
            
        return response
    
    raise Exception("Max retries exceeded")
```

### 2. Распределяйте запросы во времени

Распределяйте вызовы API, а не отправляйте их рывками:

- Запускайте пакетные задачи с разными интервалами
- Добавляйте задержки между запросами при обработке больших наборов данных
- Используйте системы очередей для сглаживания пиков нагрузки

### 3. Используйте кэширование

**Для Analytics API и AI Code Tracking API:**

Эти API поддерживают HTTP-кэширование с ETag’ами (ETags). См. раздел [Caching](#caching) выше, чтобы узнать, как использовать ETag’и для снижения трафика и избегания лишних запросов.

**Ключевые преимущества:**

- Меньше трафика
- Более быстрые ответы, если данные не изменились
- Не учитывается в лимитах запросов (для ответов 304)

Используйте краткие обозначения дат (`7d`, `30d`) вместо меток времени для лучшей поддержки кэширования в Analytics API.

### 4. Отслеживайте использование

Отслеживайте шаблоны запросов, чтобы не выходить за пределы лимитов:

- Записывайте отметки времени и коды ответов API-вызовов
- Настройте оповещения на ответы 429
- Отслеживайте ежедневные и недельные показатели использования
- Корректируйте интервалы опроса по фактическим потребностям

### 5. Объединяйте запросы разумно

Для конечных точек с пагинацией:

- Выбирайте подходящий размер страницы, чтобы получать больше данных за один запрос
- Для конечных точек Analytics API по пользователям: используйте параметр `users` для фильтрации конкретных пользователей
- Для больших выгрузок данных: используйте конечные точки CSV, когда они доступны (они эффективно передают данные потоково)

### 6. Опрос с разумными интервалами

Не опрашивайте слишком часто эндпоинты, которые обновляются редко:

- **Admin API** `/teams/daily-usage-data`: Опрашивайте не чаще одного раза в час (данные агрегируются по часам)
- **Admin API** `/teams/filtered-usage-events`: Опрашивайте не чаще одного раза в час (данные агрегируются по часам)
- **Analytics API**: Используйте сокращённые диапазоны дат (`7d`, `30d`) для лучшей поддержки кэширования
- **AI Code Tracking API**: Данные поступают почти в реальном времени, но опрос раз в несколько минут достаточен

### 7. Корректно обрабатывайте ошибки

Реализуйте правильную обработку ошибок для всех вызовов API:

```
async function fetchAnalytics(endpoint) {
  try {
    const response = await fetch(`https://api.cursor.com${endpoint}`, {
      headers: {
        'Authorization': `Basic ${btoa(API_KEY + ':')}`
      }
    });
    
    if (response.status === 429) {
      // Rate limited - implement backoff
      throw new Error('Rate limit exceeded');
    }
    
    if (response.status === 401) {
      // Invalid API key
      throw new Error('Authentication failed');
    }
    
    if (response.status === 403) {
      // Недостаточно прав доступа
      throw new Error('Enterprise access required');
    }
    
    if (!response.ok) {
      throw new Error(`API error: ${response.status}`);
    }
    
    return await response.json();
  } catch (error) {
    console.error('API request failed:', error);
    throw error;
  }
}
```

## Общие ответы об ошибках

Все API используют стандартные коды статусов HTTP:

### 400 Неверный запрос

Параметры запроса недействительны или отсутствуют обязательные поля.

```
{
  "error": "Неверный запрос",
  "message": "Некоторые пользователи не состоят в команде"
}
```

### 401 Несанкционированный

Недействительный или отсутствующий ключ API.

```
{
  "error": "Unauthorized",
  "message": "Неверный API-ключ"
}
```

### 403 Доступ запрещён

API-ключ действителен, но прав доступа недостаточно (например, для функций Enterprise при тарифе не Enterprise).

```
{
  "error": "Forbidden",
  "message": "Enterprise access required"
}
```

### 404 Не найдено

Запрошенный ресурс не существует.

```
{
  "error": "Не найдено",
  "message": "Ресурс не найден"
}
```

### 429 Слишком много запросов

Превышен лимит запросов. Используйте экспоненциальный бэкофф.

```
{
  "error": "Слишком много запросов",
  "message": "Превышен лимит запросов. Попробуйте позже."
}
```

### 500 Внутренняя ошибка сервера

Ошибка на стороне сервера. Обратитесь в поддержку, если проблема не исчезает.

```
{
  "error": "Внутренняя ошибка сервера",
  "message": "Произошла непредвиденная ошибка"
}
```
