# Контекст
- Цель: перестроить `cursor_api/file_agent` в личный CLI-пайплайн для перевода больших текстовых файлов через параллельные локальные Cursor Agent запуски модели `composer-2.5`.
- Основной рабочий сценарий:
  - пользователь просит модель обработать файл, например `sources/15/test.txt`;
  - модель запускает из `cursor_api`: `python -m file_agent run sources/15/test.txt -t`;
  - CLI нормализует исходный файл, делит его на части по границам предложений и сохраняет части в ту же папку;
  - первая часть остается в исходном файле `test.txt`;
  - остальные части создаются рядом с суффиксами текущего формата `testS001.txt`, `testS002.txt`, ...;
  - каждая часть параллельно отправляется отдельному Cursor Agent;
  - агент переводит часть на русский и упрощает текст по промпту из `cursor_api/file_agent/prompt.md`;
  - ответ агента заменяет содержимое того же файла части;
  - CLI ждет все параллельные части, сохраняет все успешные результаты и только потом возвращает общий статус.
- Второй рабочий сценарий:
  - после ручной проверки частей пользователь просит собрать файл;
  - модель запускает из `cursor_api`: `python -m file_agent run sources/15/test.txt -merge`;
  - CLI находит `test.txt` + `testS001.txt`, `testS002.txt`, ... в порядке номеров и перезаписывает исходный `sources/15/test.txt` объединенным текстом.
- Текущий код:
  - `cursor_api/file_agent/cli.py` сейчас требует `-t/--task` с текстом задачи, это нужно заменить на флаг режима перевода.
  - `cursor_api/file_agent/runner.py` уже запускает локального агента через `cursor-sdk-bridge` и модель `composer-2.5`, но обрабатывает файлы как обычную задачу.
  - `cursor_api/file_agent/normalization.py` умеет нормализовать текст в предложения и мерджить список файлов, но его контракт не совпадает с новым CLI.
  - `cursor_api/file_agent/split.py` содержит старый самостоятельный скрипт разбиения, его нужно адаптировать в модуль пайплайна.
  - `cursor_api/file_agent/prompt.md` содержит единственный редактируемый пользователем промпт перевода.
  - `cursor_api/file_agent/keys.py` читает API-ключ из внешнего текстового файла. Этот файл и путь не менять.
  - `cursor_api/README.md` сейчас короткая шпаргалка, ее нужно заменить подробной инструкцией пользователя по новому процессу.
- Архитектурные решения пользователя:
  - Не использовать бэкапы.
  - Не использовать фоллбеки.
  - Не переключаться на другие модели.
  - Не добавлять скрытые запасные режимы.
  - Ошибка должна быть ошибкой, а не автоматической попыткой угадать другой путь.
  - Модель для работы только `composer-2.5`.
  - Ключ хранится только во внешнем текстовом файле, `keys.py` не трогать без явного запроса.
  - На каждую попытку агента отводится 60 секунд.
  - На каждую часть максимум 3 попытки.
  - Если часть упала, остальные части все равно должны продолжать работу.
  - Успешные части должны быть сохранены даже при общем ошибочном завершении.
  - После всех попыток CLI должен кратко вывести, какие файлы успешны и какие упали.
  - Проверка размера: считать символы без пробелов по каждой исходной части до перевода и после перевода; допустимый диапазон `60%..140%`.
  - Деление: ориентир `150` строк на часть после нормализации и примерно `5000` символов на часть; границы частей должны идти по предложениям.
  - `prompt.md` нужно использовать как system prompt. Если при тестировании SDK явно не поддержит такой формат, не делать скрытый фоллбек, а остановиться с ошибкой и согласовать переход на обычный user prompt отдельно.
- Решение по служебным файлам:
  - Не добавлять manifest, sidecar, backup или скрытые временные копии.
  - Порядок merge восстанавливать по имени исходного файла и суффиксам `S001`, `S002`, ...
  - Исходные размеры частей для проверки держать в памяти в рамках текущего запуска `-t`.
  - Консольный вывод использовать как главный отчет для пользователя.
- Работать только внутри:
  - `cursor_api/README.md`
  - `cursor_api/file_agent/cli.py`
  - `cursor_api/file_agent/runner.py`
  - `cursor_api/file_agent/normalization.py`
  - `cursor_api/file_agent/split.py`
  - новые модули внутри `cursor_api/file_agent/`, если они реально подключены к CLI
  - тесты или локальные проверочные скрипты внутри `cursor_api/`, если нужны для доказательства результата
- Не трогать:
  - `cursor_api/file_agent/keys.py`
  - внешний файл ключа
  - пользовательские файлы `sources/*` во время реализации, кроме осознанной ручной проверки на копии или специально созданном тестовом файле

## Критерий завершения

Работу можно считать законченной, когда одновременно верны все пункты:

1. `python -m file_agent run <file> -t` запускает полный режим: нормализация → разбиение → параллельный перевод частей → сохранение успешных частей → проверка размера → краткая сводка.
2. `python -m file_agent run <file> -merge` собирает `<file>` + `<name>S001<ext>` + `<name>S002<ext>` в исходный `<file>` в правильном порядке.
3. `-t` больше не требует текст задачи и всегда читает инструкцию только из `cursor_api/file_agent/prompt.md`.
4. Каждая часть обрабатывается отдельным агентом `composer-2.5`, с лимитом `60` секунд на попытку и максимум `3` попытками на часть.
5. Ошибка одной или нескольких частей не отменяет ожидание и сохранение остальных успешных частей.
6. После обработки CLI завершает процесс с кодом `0`, только если все части успешно обработаны и прошли проверку `60%..140%` по символам без пробелов.
7. `cursor_api/README.md` содержит точную пользовательскую инструкцию: как запускать перевод, как читать ошибки, как вручную править части и как запускать merge.
8. Минимальная локальная проверка подтверждает разбиение, merge, проверку размеров и поведение при частичной ошибке без реального расхода API, а реальный агентский запуск оставлен отдельной ручной проверкой пользователя.
